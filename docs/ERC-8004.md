# ERC-8004 Integration — Feasibility Report & Plan

> Spike requested after the ERC-8126/8004/8183 agent-trust stack surfaced (June 2026).
> Question: should RugSense plug into ERC-8004, what does it cost, and how?
> **Verdict: yes for the two cheap levels (now), defer the expensive one (build on trigger).**

## What ERC-8004 actually is (verified against the spec + contracts repo)

A live, adopted on-chain trust layer for AI agents — **not vapor**. Three registries,
deployed as per-chain singletons with the same vanity addresses across 16+ EVM chains
**including Base (chainId 8453)**:

| Registry | Address (same on Base + mainnet) | What it does |
|----------|----------------------------------|--------------|
| Identity | `0x8004A169FB4a3325136EB29fA0ceB6D2e539a432` | ERC-721; `register(agentURI)` mints an agent NFT (`agentId`). `agentURI` → off-chain `/.well-known/agent-card.json`. |
| Reputation | `0x8004BAa17C55a88189AE136b182e5fdA19dE9b63` | Any non-owner calls `giveFeedback(agentId, value 0-100, …, feedbackURI)`. Third-party attestations about an agent. |
| Validation | (per-chain singleton) | An agent calls `validationRequest(validator, agentId, requestURI, hash)`; the validator answers `validationResponse(requestHash, response 0-100, …)`. **Response is 0-100 — our exact score range.** |

Adoption signals: ERC-8004 live on mainnet since 29 Jan 2026; **150k–247k agents
registered** across chains (8004scan.io / RNWY); tooling exists (`npx create-8004-agent`,
Agent0/Azeth SDKs, 8004scan explorer, Agent0 subgraph); RedStone+Credora already supply
risk intelligence into the stack. On Base, gas is **"well under a cent"** — registration
is one cheap tx, attestations are economically viable.

## The honest fit

- ERC-8126 verifies an **agent's own** token/code/wallet — NOT arbitrary tradeable
  tokens. So 8126 is **not a literal slot** for "score this random launch." Don't overclaim.
- What DOES fit RugSense, literally:
  1. **Identity** — register RugSense as a discoverable service-agent. Pure upside: a new
     discovery channel (the 150k-agent registry + its crawlers) beyond Bazaar/Agentic.Market.
  2. **Validation** — `validationResponse` is a 0-100 score. RugSense can act as a
     **token-risk validator**: an agent posts a validation request naming our validator
     address + the token; we answer 0-100 on-chain. This is the real product slot — but it
     needs an on-chain responder + gas per response, so it's only worth building once
     agents actually send requests.
  3. **Reputation** — our verifiable track record is exactly the "which provider do you
     trust" signal that reputation aggregators consume. Selection in a provider marketplace
     rewards measurable accuracy → doubles down on what we already built.

## Recommendation

- **Now (free):** ship Level 1 — `public/.well-known/agent-card.json`. Zero gas, zero risk,
  instant discoverability. (Shipped with this report.)
- **Soon (cents):** run Level 2 when you have a Base wallet funded with a little ETH for
  gas. `scripts/register-8004.ts` is written and env-gated — it does NOT run without
  `REGISTER_PRIVATE_KEY`. After it prints your `agentId`, paste the `registrations` entry
  into the agent-card and redeploy. (Use a low-value hot wallet, not the receiving wallet.)
- **Later (defer):** Level 3 validator endpoint — only after trading agents start reading
  per-token validation/attestations pre-swap. That's the trigger; until then it's gas +
  infra with no consumer. Keep our scoring output attestation-shaped (already 0-100 +
  structured `checks[]`) so the lift stays small when the trigger fires.

## Where this slots into the existing plan

Add to `docs/AGENT-DEMAND-RESEARCH.md` as a distribution lane (alongside Agentic.Market
